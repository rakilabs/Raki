# Raki Project Memory — Spec Review Lessons

> Accumulated lessons from spec reviews so we stop re-learning the same quirks.

## 2026-06-13 — TipTap Block IDs Spec Review

### Known state that every spec must account for

- **Body-format changes are IPC-contract changes.** Both input DTOs (`CreateNoteInput`, `UpdateNoteInput`) and output DTOs (`NoteDto`) must be updated together with the command implementations that convert between them. A spec that only updates one side will corrupt data.
- **`content_hash` covers raw `(title, body)`.** Any change to the body format (plain text → ProseMirror JSON, stable block IDs, etc.) invalidates `embedded_hash` and forces a full-corpus re-embed. The spec must state the invalidation/backfill strategy.
- **Block-level provenance requires the full chain.** Stable block IDs in the editor are not enough; the spec must also show how IDs flow through `Block`, `body_to_blocks`, chunk source IDs, and vector IDs. Otherwise edits still delete all note vectors.
- **Frontend normalization violates AGENTS.md §5.** Body normalization and ID assignment belong in `raki-domain` / backend commands, not in components.
- **Clone operations duplicate ProseMirror node attributes.** Any block-ID extension must include a deduplication pass for copy-paste, drag-drop, and undo/redo.

### Recurring spec gaps to watch for

- Editor-swap specs must audit all `NoteDto.body` consumers, regenerate IPC bindings, and require `bun run typecheck` green.
- Autosave/debounce specs must define in-flight save serialization, optimistic-concurrency handling, and error UX.
- JSON-body validation must specify allow-listed node types, byte/block limits, and malformed-body behavior (reject vs normalize).

## 2026-06-12 — R4 Memory Lifecycle Signals Spec Review

### Known state that every spec must account for

- **`hybrid_search` currently returns `Vec<NoteId>` without scores.** Any spec proposing a multiplicative mixer over `retrieval_score` must first introduce a scored retrieval primitive (e.g., `Vec<ScoredNote>`) and state the score-normalization rule.
- **`SignalSource` is read-only; there is no `SignalStore` write port.** Any spec that mutates signals (view counts, last accessed, pin) from a Tauri command must add a write-side domain port and adapter, or the command will leak storage concerns into `raki-app`.
- **Every new user-data table must carry the sync-ready row conventions.** `created_at`, `updated_at`, `deleted_at`, and `version` are mandatory (AGENTS.md §7 / ADR-0002). A table keyed to a soft-deleted parent must cascade/purge or inherit `deleted_at` to avoid leaking behavioral metadata.
- **`raki-retrieval` may only depend on `raki-domain`.** A spec that shows `raki-retrieval` calling a concrete function/type in `raki-memory` is a dependency-rule violation. The contract must live in `raki-domain`.

### Recurring spec gaps to watch for

- Measurement-gated milestones must state automated pass/fail rules. "Positive and statistically meaningful" is unverifiable. Specify metric, delta, statistical test, number of runs, and max tuning iterations.
- Reproducible eval corpora must pin their configuration. Provider, model id+version, chunking algorithm version, RRF parameters, and random seed must be recorded alongside any baseline; regeneration triggers must be enumerated.
- Any new Tauri command that mutates user-behavior state needs explicit atomicity, idempotency, debounce/throttle, tombstone handling, and trust-boundary spec. Frontend-driven writes are not implicitly safe.
- Behavioral tracking metadata (view counts, access times) is a PII surface even when aggregated. Every spec must address locality, retention, export, user deletion, and inclusion in backups/exports.
- Eval corpora targeting personal domains (finance, health, trips, people) must be explicitly synthetic/anonymized with a PII review checklist and `DATA_PROVENANCE.md`.
- Multiplicative score mixers must specify input validation and observability. Validate `f64` inputs (`is_finite()`, positive half-life, non-negative weights) and return per-signal contribution breakdowns for debugging.
- Important new domain concepts and policies require an ADR. If a spec introduces new ports, tables, behavioral logging policies, or measurement-gated splits, it must commit to authoring the ADR.

## 2026-06-08 — Slice 3 Note Body Editor Review

### Known state that every spec must account for

- **All existing notes have `body: "{}"`** (hardcoded by `create_note`). Any body-format change must include a legacy-data migration or special-case, or the first user edit will corrupt empty notes into notes containing literal `"{}"`.
- **`raki-domain` does not currently depend on `serde_json`**. Any spec proposing JSON parse/serialize logic in the domain kernel must explicitly call out the dependency addition.
- **`content_hash` in `raki-storage` is computed over raw `(title, body)`**, not over flattened text. Any change to the text flattener (space-join → newline-join, etc.) will not trigger re-embedding via hash mismatch. The spec must include an explicit embedding invalidation strategy (clear `embedded_hash`, version the flattener, etc.).
- **`update_note` is currently a non-atomic read-modify-write** (`get` → edit in memory → `upsert`). Any edit command spec should consider the TOCTOU window: concurrent soft-delete can resurrect a tombstoned note unless the UPDATE has a `deleted_at IS NULL` guard or runs inside the same transaction as the read.

### Recurring spec gaps to watch for

- **Migration / backfill for index changes:** When the FTS5 or vector indexing contract changes, the spec must state how existing rows are backfilled. "No existing notes have real content" is only true until the slice ships — after that, the assumption is invalid for the next slice.
- **Validation boundaries:** The `Note` entity has no validation rules in the spec. Empty titles, empty bodies, and oversize inputs need explicit constraints or UI fallbacks.
- **Defensive JSON flattening:** `body_to_text` runs on every read path. The spec must define total behavior for structurally valid but unexpected PM JSON (missing `content`, unknown node types, marks), not just "non-doc fallback."

## 2026-06-08 — R0 SciFact Benchmark Tier Spec Review

### Known state that every spec must account for

- **`ndcg_at_k` in `raki-retrieval/src/metrics.rs` is already graded** — it accepts `grades: &HashMap<String, f64>` and computes graded DCG / ideal-DCG. Any spec proposing a new "graded" nDCG primitive must first verify the existing signature; the current function is usable for BEIR qrels directly.
- **`raki-eval` already contains an `eval-report` binary with a `--write` flag** before persisting artifacts. Any new binary in the same crate that writes to the working tree should respect that CLI contract.
- **The `run_eval_over` in-memory index construction pattern** (`Database::open_in_memory()` + repos + `register_sqlite_vec()`) is already duplicated in spirit across eval paths. New eval tiers should extract a shared helper rather than sanction duplication.

### Recurring spec gaps to watch for

- **"Untouched" crate lists must be cross-checked against the components-touched list (D7).** A direct contradiction between "Is NOT" and "Components touched" is a CRITICAL finding.
- **Benchmark/eval DoDs must define actionable thresholds.** "Recorded delta to act on" is unverifiable without a sign or magnitude rule. Every measurement-gated milestone must state what number constitutes go/no-go.
- **Download-and-cache specs must address integrity and concurrency.** "Fetch once; reuse thereafter" needs atomic extraction, validation, and a recovery path for partial writes, or the benchmark can silently consume corrupt data.

## 2026-06-10 — R1 Reranker Attach Spec Review

### Recurring spec gaps to watch for

- **Local model inference on user-facing paths needs timeout, cancellation, and `spawn_blocking` boundaries.** "Degrade-on-error" (returning `Err`) is insufficient for hangs, deadlocked ONNX runtimes, or CPU thrashing. Any spec wiring a local model into production must explicitly state the timeout and how CPU-bound work is kept off the async runtime.
- **Materializing user content for model consumption needs size bounds and truncation.** Feeding unbounded ProseMirror JSON → text into a cross-encoder causes memory spikes (POOL × large bodies) and silent model truncation. Every spec that calls `body_to_text` for model input must define a per-candidate limit.
- **"Attach-to-validate" specs must define observability up front.** Deferring metrics, latency SLOs, and fallback telemetry to a later evaluation phase leaves the kill-switch decision arbitrary. Any spec whose premise is "attach now, measure later" must include the instrumentation requirements in the same slice.

## 2026-06-10 — R2 Chunking Baseline Record Spec Review

### Known state that every spec must account for

- **`bench.rs` patterns are not sufficient spec for working-tree writes.** Mirroring an existing binary's `--write` flag does not automatically cover atomicity, overwrite policy, parent-directory creation, or path resolution. Any new `--write`-style artifact must state those explicitly rather than assume the prior pattern is safe.
- **`eval-data/real` and any stdout path that can emit user notes is a PII surface.** Even when gitignored and never committed, stdout can be captured by shell history, terminal scrollback, CI logs, or `script`. The spec must state the privacy/egress decision for any real-user-data path.

### Recurring spec gaps to watch for

- **Critical-path dependencies must be scoped before they become gates.** When a spec declares another slice (e.g., "P1") the explicit next critical-path item, that dependency itself needs scope, deliverables, acceptance criteria, owner, timeline, and a privacy/consent model in the same or a referenced spec. Otherwise the downstream verdict is unbounded.
- **Generated artifact specs need regeneration triggers.** A baseline or report that is "run once" becomes stale the moment upstream fixtures, formatters, or metrics change. The spec should state when it must be regenerated and who owns the regeneration.
- **Embedding/real-model baselines need reproducibility stance.** If the winning arm or metric values depend on a real model, the spec must either pin provider/model/version/seed or explicitly disclaim the baseline as configuration-dependent and directional only.

## 2026-06-10 — R2 Chunking Production Migration Spec Review

### Known state that every spec must account for

- **`VectorIndex` is a generic domain port** (not note-specific). Any addition like `delete_by_note` is a boundary violation. Use `delete_by_prefix` or a wrapper in the service layer instead.
- **`content_hash` covers raw `(title, body)` only.** It does NOT capture chunking algorithm version, prefix format, or cap threshold. Any spec changing the chunking pipeline must include an embedding invalidation strategy (e.g., `embedding_alg_version` column or model-id suffix).
- **`list_pending` currently has no `ORDER BY`.** Any spec that triggers a full-corpus re-index must explicitly state prioritization (e.g., `updated_at DESC`) and an expected SLO, or recent notes will starve.
- **`embed_one` was designed for 1:1 note:vector semantics.** Moving to 1:N chunk semantics requires revisiting batch bounds, atomicity, and timeout assumptions. "Milliseconds" claims about transient states are invalid when N round-trips are involved.

### Recurring spec gaps to watch for

- **Destructive migrations before measurement gates.** Any spec ordering `DROP TABLE` or data destruction before a binding quality verdict is a CRITICAL red flag. The default assumption must be dual-schema or feature-flag coexistence during evaluation.
- **"Zero-trait-changes" claims must be verified against actual diffs.** Specs sometimes claim a change is "zero-overhead, zero-trait-changes" while literally adding methods to traits. Call out the contradiction.
- **Additive retrieval layers need feature flags by default.** Shipping an unmeasured enhancement (e.g., contextual prefixing) as the production default bypasses ADR-0007. The spec must explicitly gate the default on its own measurement.
- **Batch boundaries must be explicit when scale changes.** A "single batch call" that was safe for 1 item becomes unsafe for N items. Specs must state max batch size, sub-batching strategy, timeout, and `spawn_blocking` boundary.
- **Eval data paths need privacy decisions in every spec.** `eval-data/real`, stdout, and CI logs are PII surfaces. Every spec touching real user data must state locality, consent, anonymization, and retention — even for "manual" tests.

## 2026-06-11 — R3 Query Understanding Spec Review

### Known state that every spec must account for

- **`raki-generate` does not exist.** Context assembly (`assemble_for`, `AssembledContext`, `GenerateDeps`) is owned by `raki-memory`. Any spec referencing `raki-generate` is referencing a non-existent crate and must be corrected to `raki-memory` or `raki-app`.
- **`AssembledContext` lives in `raki-memory`.** Any cloud call that claims to require an `AssembledContext` for egress approval must be checked against the dependency graph. `raki-ai` cannot import `raki-memory`; the orchestration must live in `raki-memory` or the domain must define a smaller audit type.

### Recurring spec gaps to watch for

- **Confidence thresholds and gates must be defined in exactly one place.** When a parse-fallback sets `confidence = 0.3` but the integration gate requires `confidence >= 0.5`, the fallback path is dead. Cross-reference all threshold values across parsing, fallback, and integration sections.
- **Header "Gated by" must trace to an exit criterion with identical enforcement.** A +0.05 MAP gate in the header that becomes a "non-negative delta" in exit criteria, with the real threshold relegated to a manual `#[ignore]` test that only needs to be "recorded," is an unenforceable gate. Every measurement-gated milestone must state an automated pass/fail rule.
- **f64 fields used as branching gates must specify clamping/validation.** Unvalidated `f64` values (NaN, Infinity, out-of-range) silently corrupt branching logic. Always specify `is_finite()` and range bounds.
- **Markdown stripping must be explicit for any JSON-from-LLM path.** LLMs frequently wrap JSON in `` ```json `` fences. A parse-fallback that feeds raw text into embeddings/keyword search will poison retrieval with syntactic garbage unless stripping is specified.
- **Frontend specification is required for any user-facing path with >100ms latency.** Loading states, cancellation tokens, progress events, TypeScript DTO changes, and IPC bindings must be specified. "Awaiting implementation plan" is not an excuse for omitting the frontend slice.
