# Raki Project Memory — Spec Review Lessons

> Accumulated lessons from spec reviews so we stop re-learning the same quirks.

## 2026-06-08 — Slice 3 Note Body Editor Review

### Known state that every spec must account for

- **All existing notes have `body: "{}"`** (hardcoded by `create_note`). Any body-format change must include a legacy-data migration or special-case, or the first user edit will corrupt empty notes into notes containing literal `"{}"`.
- **`raki-domain` does not currently depend on `serde_json`**. Any spec proposing JSON parse/serialize logic in the domain kernel must explicitly call out the dependency addition.
- **`content_hash` in `raki-storage` is computed over raw `(title, body)`**, not over flattened text. Any change to the text flattener (space-join → newline-join, etc.) will not trigger re-embedding via hash mismatch. The spec must include an explicit embedding invalidation strategy (clear `embedded_hash`, version the flattener, etc.).
- **`update_note` is currently a non-atomic read-modify-write** (`get` → edit in memory → `upsert`). Any edit command spec should consider the TOCTOU window: concurrent soft-delete can resurrect a tombstoned note unless the UPDATE has a `deleted_at IS NULL` guard or runs inside the same transaction as the read.

### Recurring spec gaps to watch for

- **Migration / backfill for index changes:** When the FTS5 or vector indexing contract changes, the spec must state how existing rows are backfilled. "No existing notes have real content" is only true until the slice ships — after that, the assumption is invalid for the next slice.
- **Validation boundaries:** The `Note` entity has no validation rules in the spec. Empty titles, empty bodies, and oversize inputs need explicit constraints or UI fallbacks.
- **Defensive JSON flattening:** `body_to_text` runs on every read path. The spec must define total behavior for structurally valid but unexpected PM JSON (missing `content`, unknown node types, marks), not just "non-doc fallback."
